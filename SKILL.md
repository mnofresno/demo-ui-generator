# Demo UI Generator

## Overview

Use this skill to generate a standalone, ready-to-run demo UI repo from JSON input.

This skill must never add demo-only UI code to the product repository. Demo UIs belong in separate lightweight repos, ideally under the user's GitHub account, with a manifest marker that identifies the project as generated by this skill.

The skill caches UI structure by normalized `ui_type`, `features`, and `style`, then injects runtime data into `data.json` so repeated requests with the same structure avoid regeneration.

## Hard Rules

1. Never modify the product repo to mount a demo inbox, demo mail client, demo OTP screen, or any other demo-only UI.
2. Always create or reuse a standalone repo for the demo.
3. Prefer lightweight repos:
   - frontend-only when static data is enough
   - frontend + tiny API only when runtime control is actually needed
4. Every generated demo repo must include `demo-ui-generator.manifest.json`.
5. Before generating a new demo repo, check whether one already exists with the same manifest `ui_key`.
6. If a demo needs real HTML from the product, consume it through the product's existing APIs or a minimal external adapter. Do not patch the product just to host the demo UI.

## Workflow

1. Read the input JSON and normalize `ui_type`, `features`, and `style`.
2. Compute a deterministic cache key from structure only. Ignore dynamic email content.
3. Derive a stable `ui_key` from the same structural payload.
4. Check whether a standalone demo repo already exists with the same `ui_key`.
5. Reuse `/cache/<ui_type>/<hash>/` when it exists; otherwise seed the cache from `templates/email-ui-base`.
6. Copy the selected template into a standalone output repo, never into the product repo.
7. Write runtime data into `data.json`.
8. Write `demo-ui-generator.manifest.json` with the structural identity of the generated UI.
9. Return the generated project path, manifest path, and cache status.

## Inputs

Expected JSON shape:

```json
{
  "ui_type": "email",
  "emails": [],
  "features": {},
  "style": "minimal",
  "source_app": "gymnerd-bot",
  "demo_slug": "lucas-romero-mail"
}
```

## Output Contract

- Output is a standalone repo/project, not a patch inside the product app.
- The generated project contains runnable app files plus `data.json`.
- The generated project contains `demo-ui-generator.manifest.json`.
- The browser UI loads `data.json` dynamically.
- The page exposes `window.demoAPI` with:
  - `addEmail(email)`
  - `clearEmails()`
  - `setOTP(code)`
  - `reloadData()`

## Files To Use

- Main generator: `scripts/generate.js`
- Base structure template: `templates/email-ui-base/`
- Structural cache: `cache/email/<hash>/`
- Verification: `tests/run-self-test.js`

## Notes

- This skill is intentionally standalone. It may feed from external APIs, but the generated demo UI must live outside the product repo.
- The cache key is based on structure, not email content.
- If you need to change the structural HTML/CSS/JS for email UIs, update `templates/email-ui-base/` and let new hashes create new cache entries.
- Use the manifest file as the primary tag to detect whether a matching demo UI already exists.
